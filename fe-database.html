🗃️ データベース

SQLと正規化は「知っていれば確実に取れる」典型分野です。科目Aでは正規化・主キー/外部キー・SQL（SELECT／集計／結合）・トランザクション（ACID・排他制御）が「よく出る」の常連なので、丸暗記ではなく仕組みから理解しておくと安心です。

5-1関係データベースの基本 — 表・行・列とDBMSの仕事

🎯 ゴール：表・行・列の呼び名と、DBMS（データベース管理システム）がやってくれていることを説明できる

関係データベース（リレーショナルデータベース）は、データをExcelの表のような「表（テーブル）」の形で保存するしくみです。表の横1行分を「行（レコード）」、縦1列分を「列（フィールド）」と呼びます。例えば名簿の表なら、1人分のデータが1行（レコード）、「氏名」や「電話番号」といった項目が1列（フィールド）にあたります。

表の呼び方意味Excelでのイメージ
表（テーブル）データのまとまり全体1枚のシート
行（レコード）1件分のデータ横1行
列（フィールド）項目の種類縦1列（列見出し）

データベースを管理する専用ソフトをDBMS（データベース管理システム）と呼びます。DBMSは、複数人が同時にアクセスしても矛盾が起きないよう制御し（同時実行制御）、途中で処理が止まっても正しい状態に戻せるようにし（障害回復）、決めたルールに反するデータが入らないよう見張る（一元管理）、という仕事をしています。Excelでも表は作れますが、複数人が同時に同じセルを編集したときの安全策や、途中で保存に失敗したときの自動復旧は基本的にありません。この「壊れない・矛盾しない仕組み」を持っているかどうかが、DBMSとただの表計算ソフトの一番の違いです。

⚡ ポイント
  • 表＝テーブル、行＝レコード、列＝フィールド。用語の言い換え問題が頻出
  • DBMSの3大仕事：一元管理・同時実行制御・障害回復

5-2主キーと外部キー — 学籍番号のように行を特定する

🎯 ゴール：主キーと外部キーの役割の違いを説明し、参照整合性の意味を答えられる

主キーとは、表の中の1行を一意に特定できる列のことです。クラス名簿でいう「学籍番号」のようなもので、同じ値が2つあってはいけません（重複不可）し、空欄（NULL）も許されません。1つの列だけで一意にできない場合は、複数の列を組み合わせて主キーにします。これを複合キーと呼びます。

外部キーとは、別の表の主キーを参照する列のことです。例えば「注文表」の中に「顧客ID」という列があり、それが「顧客表」の主キーである顧客IDを指している、というような関係です。この仕組みのおかげで、注文表には必ず実在する顧客IDしか登録できなくなります。これを参照整合性と呼びます。

⚡ ポイント
  • 主キー＝一意・NULL不可。1列で無理なら複合キーで対応
  • 外部キー＝他表の主キーを指す列。参照先に無い値は登録できない（参照整合性）

5-3正規化 — 二重持ちをなくして矛盾を防ぐ（頻出図解）

🎯 ゴール：正規化の目的と第1〜第3正規形それぞれで何をするかを説明し、簡単な例でどの正規形か判定できる

正規化とは、1つの表に情報を詰め込みすぎて同じ内容が何度も出てくる状態（二重持ち）を解消し、複数の表に分けていく作業です。目的はただ1つ、「更新したときに矛盾が起きないようにする」ことです。例えば顧客の住所を1か所にしか持たなければ、住所変更は1行直すだけで済みますが、あちこちの表に住所が重複していると、直し忘れた箇所だけ古い住所のままになってしまいます。

正規化は段階を踏んで進めます。第1正規形は、1つのセルに複数の値が入っている「繰り返し項目」をなくし、1行1件のきれいな表にする段階です。第2正規形は、複合キーの一部（キーの一部分）にしか従属していない列を、別の表に分ける段階です。第3正規形は、主キー以外の列に従属している列（主キーを経由した間接的な従属＝推移的関数従属）を、さらに別の表に分ける段階です。

✏️ 例題（本試験風） ある注文明細表は「注文番号＋商品コード」を主キーとしており、列「商品名」は商品コードのみに従属し、注文番号には従属していない（主キーの一部にしか従属していない）。この表は第何正規形の条件を満たしていないか。
答えを見る 第2正規形。「商品名」は複合キーの一部（商品コード）だけに従属する部分関数従属になっており、第2正規形（主キー全体に従属する）の条件に反している。商品名は商品表として切り出す必要がある。
⚡ ポイント
  • 正規化の目的は「二重持ちの解消」→「更新時の矛盾防止」のひとつだけ
  • 第1＝くり返し排除、第2＝キーの一部への従属を分離、第3＝キー以外への従属（推移的関数従属）を分離
  • 迷ったら「この列は主キーの全部に従属しているか？」を自問する

5-4SQL① 基本のSELECT — 表から条件で取り出す

🎯 ゴール：SELECT〜FROM〜WHEREの基本構文と、LIKE・BETWEEN・IN・DISTINCT・ORDER BYの意味を理解し、簡単なSQL文の実行結果を答えられる

SQLは、データベースに「この条件のデータをちょうだい」とお願いするための言葉です。基本の形は SELECT 列 FROM 表 WHERE 条件で、「どの表の」「どの列を」「どんな条件で」取り出すかをこの順番で書きます。ほかによく使う道具に、あいまい検索のLIKE '%文字%'（部分一致）、範囲指定のBETWEEN、複数候補のIN、重複を取り除くDISTINCT、並べ替えのORDER BYがあります。

次の「社員表」を例に、実際のSQL文と結果を対応させて見てみましょう。

社員ID氏名部署年齢給与
1佐藤営業34280000
2鈴木開発28320000
3高橋営業45350000
4田中開発31300000
5伊藤総務52260000

実行前の表：社員表（上のとおり5行）

SELECT 氏名, 部署, 年齢 FROM 社員 WHERE 部署 = '開発' ORDER BY 年齢

結果の表：WHEREで「部署＝開発」に絞り、ORDER BYで年齢の小さい順に並べ替え

氏名部署年齢
鈴木開発28
田中開発31

もし氏名に「藤」を含む人を探すなら WHERE 氏名 LIKE '%藤%'（佐藤・伊藤の2件がヒット）、年齢30〜50歳の範囲なら WHERE 年齢 BETWEEN 30 AND 50、部署が営業か総務のどちらかなら WHERE 部署 IN ('営業','総務')のように書きます。

⚡ ポイント
  • SELECT-FROM-WHEREの順番と役割をセットで覚える
  • LIKE＝あいまい、BETWEEN＝範囲、IN＝候補の列挙、DISTINCT＝重複除去
  • ORDER BYを付けない限り、結果の行の並び順は保証されない

5-5SQL② 集計 — GROUP BYとHAVINGの違い

🎯 ゴール：COUNT/SUM/AVG/MAX/MINの集計関数とGROUP BYの動きを理解し、WHEREとHAVINGの使い分けを説明できる

集計関数は、複数の行をまとめて1つの値にする道具です。COUNT＝件数、SUM＝合計、AVG＝平均、MAX／MIN＝最大値・最小値を求めます。GROUP BYを使うと、指定した列の値ごとにグループを作り、グループごとに集計できます。「部署ごとの平均給与」のような集計は、GROUP BYの定番の使い方です。

ここで定番のひっかけが、WHEREとHAVINGの違いです。WHEREは集計する前の「行」に対する条件（この行を集計に含めるかどうか）、HAVINGは集計した後の「グループ」に対する条件（このグループを結果に残すかどうか）です。順番も「行を絞る(WHERE)→グループ化する(GROUP BY)→グループを絞る(HAVING)」の順で動きます。

実行前の表：5-4と同じ社員表（5行）

SELECT 部署, COUNT(*) AS 人数, AVG(給与) AS 平均給与 FROM 社員 GROUP BY 部署 HAVING COUNT(*) >= 2

結果の表：部署ごとにグループ化し、人数2人以上のグループだけ残す（総務は1人なので除外）

部署人数平均給与
営業2315000
開発2310000
✏️ 例題（本試験風） 上の社員表に対して SELECT 部署, COUNT(*) FROM 社員 GROUP BY 部署 HAVING COUNT(*) >= 2 を実行すると、総務のグループは結果に含まれるか。
答えを見る 含まれない。総務は伊藤の1人だけでCOUNT(*)=1となり、HAVING COUNT(*)>=2の条件を満たさないため、集計後にグループごと除外される。営業（2人）と開発（2人）だけが結果に残る。
⚠️ 注意 WHEREは「行」への条件、HAVINGは「集計後のグループ」への条件。HAVINGの条件にCOUNTやAVGなどの集計関数を書けるのに対し、WHEREの段階ではまだ集計されていないので集計関数は使えない。

5-6SQL③ 結合 — 内部結合と左外部結合

🎯 ゴール：内部結合と左外部結合の違いを説明し、簡単な2表からそれぞれの結合結果の行数を求められる

結合（JOIN）は、複数の表を関連づけて1つの表のように扱うための操作です。内部結合（INNER JOIN）は、両方の表に共通するキーを持つ行だけを残します。片方にしか無いデータは結果から消えます。左外部結合（LEFT OUTER JOIN）は、左側の表の行を全部残し、対応する右側のデータが無い場合はその部分をNULL（値なし）で埋めます。

実行前の表：社員表（4行・田中は部署未配属でNULL）と部署表（3行・総務にはまだ誰も配属されていない）

社員ID氏名部署ID
1佐藤10
2鈴木20
3高橋10
4田中NULL
部署ID部署名
10営業
20開発
30総務
SELECT 社員.氏名, 部署.部署名 FROM 社員 INNER JOIN 部署 ON 社員.部署ID = 部署.部署ID

結果の表（内部結合）：田中（部署ID無し）と総務（該当社員なし）は結果に出てこない

氏名部署名
佐藤営業
鈴木開発
高橋営業
SELECT 社員.氏名, 部署.部署名 FROM 社員 LEFT JOIN 部署 ON 社員.部署ID = 部署.部署ID

結果の表（左外部結合）：左表（社員）は4人全員残り、田中の部署名はNULLになる

氏名部署名
佐藤営業
鈴木開発
高橋営業
田中NULL
✏️ 例題（本試験風） 上の社員表（4行）と部署表（3行）で、内部結合の結果は何行になるか。また左外部結合の結果は何行になるか。
答えを見る 内部結合＝3行（佐藤・鈴木・高橋。部署未配属の田中と、社員のいない総務は対象外）。左外部結合＝4行（社員表の4人全員が残り、田中の部署名はNULL）。
⚡ ポイント
  • 内部結合＝両方に存在する行だけ。片方にしかないデータは消える
  • 左外部結合＝左表は全部残す。対応が無ければNULLで埋める

5-7トランザクションとACID — 銀行振込で考える原子性

🎯 ゴール：トランザクションの意味とACID特性（特に原子性）を、振込処理を例に説明できる

トランザクションとは、「ここからここまでを1つのまとまりとして扱う処理」のことです。代表例が銀行振込です。振込処理は「Aさんの口座から引き落とす」処理と「Bさんの口座に入金する」処理の2つで構成されますが、この2つは必ずセットで成功するか、セットで失敗するかのどちらかでなければなりません。引き落としだけ成功して入金が失敗する、という中途半端な状態は絶対に許されないのです。

この「全部やるか、全部やらないか」という性質を原子性（Atomicity）と呼びます。トランザクションが正常に完了して結果を確定させることをコミット、途中で異常が起きて処理前の状態に戻すことをロールバックと呼びます。トランザクションが守るべき性質は原子性を含めて4つあり、頭文字を取ってACIDと呼びます。

頭文字性質意味
A原子性（Atomicity）全部実行するか、全く実行しないかのどちらか
C一貫性（Consistency）処理の前後でデータの整合性ルールが保たれる
I独立性（Isolation）複数のトランザクションが同時実行されても互いに影響しない
D耐久性（Durability）コミットした結果は障害が起きても失われない
⚡ ポイント
  • トランザクション＝「ひとまとまり」の処理。振込＝引落＋入金がセット
  • コミット＝確定、ロールバック＝取り消し（処理前に戻す）
  • ACID＝原子性・一貫性・独立性・耐久性の4つ

5-8同時実行制御（ロック） — 共有ロックと専有ロック

🎯 ゴール：共有ロックと専有ロックの違い、DBでもデッドロックが起こることを説明できる

複数の人が同時に同じデータを読み書きすると、更新内容が消えてしまうなどの事故が起こります。これを防ぐために、DBMSはデータに「鍵（ロック）」をかけます。共有ロックは読み取り用の鍵で、複数の人が同時にかけることができます（みんなで見るだけなら問題ないため）。専有ロック（排他ロック）は書き込み用の鍵で、これがかかっている間は他の人は読むことも書くこともできません。

ロックをかける範囲の細かさをロックの粒度と呼びます。行単位のように細かくロックすると、他の人が別の行を触れるので同時に処理できる量（並行性）は上がりますが、管理するロックの数が増えて負荷が重くなります。逆に表単位のように粗くロックすると、管理は楽になりますが同時に処理できる量は下がります。

ロックは、コンピュータのOSで学んだ排他制御と同じ考え方です。そのため、DBの世界でもデッドロック（お互いが相手の持つロックの解放を待ち続けて、永久に処理が進まなくなる状態）が起こり得ます。対策として、複数のトランザクションがロックを取得する順序を統一する、といった方法が使われます。

⚡ ポイント
  • 共有ロック＝読み取り用・複数OK、専有ロック＝書き込み用・独占
  • ロックの粒度が細かい＝並行性は上がるが管理コストも上がる
  • デッドロックはDBでも起こる。ロック取得順序の統一で防ぐ

5-9障害回復 — ロールバックとロールフォワードの使い分け（頻出図解）

🎯 ゴール：ログ（ジャーナル）の役割と、障害の種類に応じてロールバックとロールフォワードのどちらを使うかを判断できる

DBMSは、データを更新するたびに「更新前の内容」と「更新後の内容」をログ（ジャーナル）という記録に残しています。このログのおかげで、障害が起きても正しい状態まで戻したり、進めたりすることができます。障害の種類によって、使う回復方法が変わります。

トランザクションの処理中に異常終了した場合（プログラムのバグや停電など、ディスク自体は無事な場合）は、ログの更新前情報を使って、その処理が始まる前の状態まで巻き戻します。これをロールバックと呼びます。一方、ディスクそのものが物理的に壊れてしまった場合は、更新前の情報も一緒に失われているので巻き戻すことができません。この場合は、以前取っておいたバックアップを戻し、そこにログの更新後情報を適用して、壊れる直前の状態まで再現します。これをロールフォワードと呼びます。

⚡ ポイント
  • 処理中の障害（ディスクは無事）→ ロールバック（更新前情報で戻す）
  • ディスク障害（データ消失）→ ロールフォワード（バックアップ＋更新後情報で進める）
  • チェックポイントを取っておくと、それ以降のログだけで済み回復が速くなる

5-10インデックスとDB応用 — 本の索引で検索を速くする

🎯 ゴール：インデックスの効果とトレードオフ、データウェアハウス・NoSQLの特徴を説明できる

インデックスとは、表の特定の列にあらかじめ「索引」を作っておき、検索を高速化する仕組みです。分厚い辞書を最初から最後まで1ページずつめくって探すのではなく、巻末の索引を使ってページ数を先に調べるようなイメージです。ただし索引にも管理コストがかかるように、インデックスを作った列は検索が速くなる代わりに、データを更新するたびにインデックス自体も更新する必要があるため、更新（追加・変更・削除）の処理は遅くなります。検索が多い列には向いていますが、更新が頻繁な列に何でもかんでもインデックスを付ければよいわけではありません。

DB応用の用語もいくつか押さえておきます。データウェアハウスは、日々の業務データを分析目的でまとめて蓄積しておく倉庫のようなものです。そこから経営判断に役立つ情報を取り出す活動をBI（ビジネスインテリジェンス）と呼びます。NoSQLは、関係データベースのような厳密な表構造にこだわらず、キーと値のペアで保存する「キーバリュー型」などを使って、規模の大きさや処理速度を優先するデータベースです。

用語特徴
インデックス検索は速くなるが、更新は遅くなる
データウェアハウス分析用にデータを蓄積する倉庫
NoSQL（キーバリュー型など）厳密さより規模・速度を優先
⚡ ポイント
  • インデックス＝検索が速くなる代わりに更新が遅くなる（トレードオフ）
  • データウェアハウス＝分析用の蓄積、BI＝そこから経営判断に活用する活動
  • NoSQL＝「厳密さより規模・速度」。関係DBとは目的が違う

🔄 振り返りクイズ（データベース）

答えを見る前に、まず口に出して答えてみるのが定着のコツ。

Q1. 表の中で1つの行を一意に特定でき、重複もNULLも許されない列を何と呼ぶか。
主キー。学籍番号のように、1行を確実に特定するための列。1列で無理なら複数列を組み合わせた複合キーにする。 → 解説 5-2 に戻る
Q2. ある注文明細表で、複合キー「注文番号＋商品コード」のうち商品コードだけに従属する列「商品名」が含まれている。この表は第何正規形の条件を満たしていないか。
第2正規形。主キーの一部（商品コード）にしか従属しない部分関数従属があるため、第2正規形の条件（主キー全体に従属すること）に反する。商品名は別表に分離する必要がある。 → 解説 5-3 に戻る
Q3. 社員表（部署：営業=2人・開発=2人・総務=1人）に対して WHERE 部署='開発' ORDER BY 年齢 を実行すると、何行がどの順で返るか（開発は鈴木28歳、田中31歳）。

Q4. 社員表（4人、うち1人は部署未配属）と部署表（3部署、うち1部署は該当社員なし）を内部結合すると何行になるか。
3行。内部結合は両方の表に一致するデータがある行だけを残すため、部署未配属の社員と、該当社員のいない部署はどちらも結果から除外される。 → 解説 5-6 に戻る
Q5. 銀行振込処理で、引き落としだけ成功し入金が失敗したときに処理全体を取り消して整合性を保つ性質は、ACIDのうちどれか。
原子性（Atomicity）。トランザクションは全部実行するか全く実行しないかのどちらかで、中途半端な状態を許さない。 → 解説 5-7 に戻る
Q6. 複数人が同時に読むことはできるが、書き込み中は他の人が読み書きできなくなるロックを何と呼ぶか。
専有ロック（排他ロック）。読み取り専用の共有ロックは複数人が同時にかけられるが、書き込み用の専有ロックは1人だけしか持てない。 → 解説 5-8 に戻る
Q7. データベースのディスクそのものが物理的に破損した場合、どちらの障害回復方法を使うべきか。
ロールフォワード。ディスク障害では更新前情報自体が失われているため、バックアップを戻したうえでログの更新後情報を適用し、最新の状態を再現する。 → 解説 5-9 に戻る
Q8. 検索を高速化できる一方、更新のたびに保守コストがかかり更新性能を落とすDBの仕組みは何か。
インデックス。本の索引と同じで、検索は速くなるが、データを更新するたびに索引自体も更新する必要があるため更新は遅くなる。 → 解説 5-10 に戻る
