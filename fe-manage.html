7-1プロジェクトマネジメントの基本 — 期限のある仕事をどう進めるか

🎯 ゴール：プロジェクトの定義とQCDのバランス、PMBOK・ステークホルダの意味を説明できるようになる。

プロジェクトとは、「期限が決まっている、一回限りの活動」のことです。毎日繰り返す通常業務（定型業務）とは違い、いつか必ず終わりが来て、同じものは二度と行われません。

プロジェクトを成功させるカギがQCD、すなわち品質（Quality）・コスト（Cost）・納期（Delivery）のバランスです。この3つはシーソーのような関係で、品質を上げようとすれば時間もお金もかかり、逆に納期を早めようとすれば品質かコストにしわ寄せがいきます。PMBOK（Project Management Body of Knowledge）は、こうしたプロジェクトの進め方を体系化した世界共通の知識ガイドです。また、ステークホルダとは発注者・利用者・開発チーム・経営層など、プロジェクトに利害関係を持つすべての人を指す言葉で、開発側だけを指すのではない点に注意しましょう。

⚡ ポイント
  • QCDは「品（Q）・コ（C）・納（D）」の3すくみ、どれか一つだけを追うと他が犠牲になる
  • ステークホルダは「利用者を含む、関係者全員」という広い範囲を問う出題に注意

7-2WBSとガントチャート — 作業を分解して見える化する

🎯 ゴール：WBSで作業を漏れなく分解し、ガントチャートで期間を読み取れるようになる。

WBS（Work Breakdown Structure：作業分解構成図）とは、プロジェクト全体を「設計」「開発」「テスト」…というように、階層的にどんどん細かい作業へ分解していく図です。引っ越し作業を「荷造り」「搬出」「搬入」…と細かいタスクに分けていくのと同じ発想で、目的は作業の抜け漏れを防ぎ、見積りの単位をそろえることにあります。

WBSで洗い出した作業を、今度は横軸に時間をとって横棒で表したのがガントチャートです。いつ始まっていつ終わるかがひと目でわかる一方、「この作業が終わらないと次が始められない」という作業同士の依存関係は表現しにくい、という弱点があります。

⚡ ポイント
  • WBSは「分解の図」、ガントチャートは「時間の図」と役割を分けて覚える
  • ガントチャートだけでは作業間の依存関係がわからない → そこを補うのがアローダイアグラム（7-3）

7-3アローダイアグラムとクリティカルパス — 全体の最短完成日数を求める

🎯 ゴール：アローダイアグラムから最早・最遅結合点時刻を計算し、クリティカルパスと所要日数を求められるようになる（毎回出る最重要計算）。

アローダイアグラム（PERT図）は、矢印（→）を「作業」、丸（◯）を「作業の区切り（結合点）」として表した図です。矢印の上に作業名と日数を書き、どの作業が終わったら次の作業が始められるかという依存関係を表します。

クリティカルパスとは、開始から終了までのすべての経路（パス）のうち、もっとも日数がかかる経路のことです。乗り換え案内で複数のルートが表示されたとき、いちばん時間のかかる乗り換えルートを思い浮かべてください。他のルートには多少の余裕（遅れても間に合う時間）がありますが、このルートだけは余裕がゼロで、少しでも遅れると到着時刻そのものが遅れてしまいます。クリティカルパスの日数が、そのままプロジェクト全体の所要日数になります。

この図の解き方を、順を追って見ていきましょう。

  1. ①最早結合点時刻（ET）を求める：開始点から順番に、その結合点に入ってくる「（手前のET）＋作業日数」をすべて計算し、矢印が複数合流する場合はその中の最大値を採用します。
  2. ②最遅結合点時刻（LT）を求める：終了点から逆向きに、その結合点から出ていく「（先のLT）－作業日数」をすべて計算し、矢印が複数枝分かれする場合はその中の最小値を採用します。
  3. ③余裕時間（TF）＝LT－ETを求める：TFが0の結合点だけをつないだ経路が、クリティカルパスです。
最早結合点時刻(ET) ＝ 合流点では max（手前のET＋作業日数）
最遅結合点時刻(LT) ＝ 分岐点では min（先のLT－作業日数）
余裕時間(TF) ＝ LT － ET　（TF＝0の経路がクリティカルパス）
結合点①②③④⑤⑥
ET（最早）04791114
LT（最遅）047101114
TF（余裕）＝LT－ET000100

結合点④だけ余裕（TF＝1）があり、それ以外の結合点はすべて余裕ゼロです。余裕ゼロの結合点だけをたどると①→②→③→⑤→⑥となり、これがクリティカルパスです。

✏️ 例題（本試験風） 上のアローダイアグラムについて、①から⑥までの最短完成日数と、クリティカルパスを求めよ。
答えを見る 方法1：経路をすべて書き出して比較する
①→②→④→⑤→⑥＝A+C+F+G＝4+5+1+3＝13日
①→②→③→⑤→⑥＝A+D+E+G＝4+3+4+3＝14日
①→③→⑤→⑥＝B+E+G＝6+4+3＝13日
3つの経路のうち最大は14日（①→②→③→⑤→⑥）。

方法2：ET・LTで検算する
ET(①)=0 → ET(②)=0+4=4 → ET(③)=max(0+6, 4+3)=max(6,7)=7 → ET(④)=4+5=9 → ET(⑤)=max(7+4, 9+1)=max(11,10)=11 → ET(⑥)=11+3=14
LT(⑥)=14 → LT(⑤)=14-3=11 → LT(④)=11-1=10 → LT(③)=11-4=7 → LT(②)=min(10-5, 7-3)=min(5,4)=4 → LT(①)=min(4-4, 7-6)=min(0,1)=0
TF＝LT－ETが0になるのは①②③⑤⑥（④だけTF＝1）。

2つの方法で一致 → 最短完成日数＝14日、クリティカルパス＝①→②→③→⑤→⑥（作業A・D・E・G）
⚡ ポイント
  • 経路を全部書き出す方法は結合点が少ないうちは確実だが、増えると大変 → ET/LTの計算のほうが速くて確実
  • クリティカルパス上の作業（TF＝0）が1日でも遅れると、プロジェクト全体がそのまま1日遅れる
  • TFに余裕がある作業（この例では④に関わるCとF）は、多少遅れても全体には影響しない

7-4工数と進捗の計算 — 人月から所要期間を見積もる

🎯 ゴール：人月の考え方で所要期間を計算し、EVMの3つの言葉の意味がわかるようになる。

人月（にんげつ）とは、「1人が1か月働いたときの作業量」を表す単位です。「20人月の仕事」といえば、1人だけでやったら20か月かかる仕事量、という意味になります。

所要期間 ＝ 工数（人月） ÷ 人数

ただし、人数を増やせば増やすほど、そのぶん単純に早く終わるとは限りません。試験問題ではこの式で単純に計算しますが、実際には新しく入った人への説明や打ち合わせの手間が増えるといった事情があり、人数を倍にしても期間がちょうど半分になるとは限らない、という点はひっかけとしても意識しておきましょう。

進捗管理でよく出てくるのがEVM（Earned Value Management）という考え方です。試験では計算までは求められず、次の3つの用語の意味を区別できれば十分です。

  • PV（Planned Value）：計画時点で予定していた出来高
  • EV（Earned Value）：実際に完了した作業の出来高
  • AC（Actual Cost）：実際にかかった費用
✏️ 例題（本試験風） 20人月と見積もられたプロジェクトを、4人で担当する場合、所要期間は何か月か。
答えを見る所要期間＝工数÷人数＝20÷4＝5か月
⚡ ポイント
  • 人数を増やしても、日数が単純に比例して減るとは限らない、というひっかけに注意
  • EVMはPV（予定）・EV（出来高）・AC（実費）の対応を問う用語問題が中心

7-5ITサービスマネジメント — ITILとインシデント管理・問題管理

🎯 ゴール：ITILの考え方と、インシデント管理と問題管理の違いを説明できるようになる。

ITIL（Information Technology Infrastructure Library）は、ITサービスを運用していくための「ベストプラクティス集」、つまり世界中の運用ノウハウをまとめた手引書のようなものです。

その中でも頻出なのが、インシデント管理と問題管理の違いです。インシデント管理は、今起きているトラブルをとにかく早く復旧させることを最優先する活動です。たとえるなら「怪我をしたときの応急処置」で、まずは血を止めることが先決、原因の究明は後回しにします。一方の問題管理は、そのインシデントがなぜ起きたのかという根本原因を突き止め、再発を防ぐための活動です。こちらは「なぜ怪我をしたのか原因を調べて、二度と同じ怪我をしないように環境を直す」ことにあたります。

このほか、システムの変更を計画的・統制的に行う変更管理、変更を実際の本番環境へ展開するリリース管理もITILの代表的なプロセスです。

⚡ ポイント
  • インシデント管理＝応急処置（復旧優先）、問題管理＝再発防止（原因究明） — この対比が定番
  • 「早く直すこと」と「二度と起きないようにすること」は別の活動だと意識する

7-6SLAとSLM — サービス水準を約束し、守り続ける

🎯 ゴール：SLAとSLMの違いを理解し、稼働率からサービス停止許容時間を計算できるようになる。

SLA（Service Level Agreement：サービスレベル合意書）とは、サービスの提供者と利用者の間で交わす「サービス水準の合意文書」です。「稼働率99.9%以上を保証します」のように、数値目標として明記され、達成できなかった場合のペナルティ条項が定められることもあります。

SLM（Service Level Management）は、SLAで約束した水準を守り続けるために、PDCAサイクルを回しながら継続的に改善していく活動です。SLAが「約束の中身」だとすれば、SLMは「その約束を守るための日々の活動」にあたります。

✏️ 例題（本試験風） SLAで稼働率99.9%と定めた場合、年間の停止許容時間はおよそ何時間か。
答えを見る 1年の時間＝365日×24時間＝8,760時間
稼働率99.9%→停止許容率＝100%－99.9%＝0.1%＝0.001
停止許容時間＝8,760時間×0.001＝8.76時間（約8.8時間）
⚡ ポイント
  • 年間時間＝365×24＝8,760時間、をまず暗記しておくと計算が速い
  • 稼働率が99.9%→99.99%と一桁上がるだけで、許容停止時間は8.76時間→0.876時間と1/10になる。桁の感覚をつかんでおく

7-7サービスデスクと運用 — 窓口の一本化と設備管理

🎯 ゴール：サービスデスクの役割とエスカレーション、ファシリティマネジメントの基本用語がわかるようになる。

サービスデスクとは、利用者からの問い合わせを受け付ける「一元窓口」のことで、SPOC（Single Point Of Contact）とも呼ばれます。問い合わせ先がバラバラだと「たらい回し」が起きてしまうため、窓口を一本化するのが目的です。サービスデスクだけでは解決できない専門的な問題は、専門部署や上位の担当者に引き継がれます。これをエスカレーションと呼びます。

ファシリティマネジメントは、建物や設備の管理を指す言葉です。代表例がUPS（無停電電源装置）で、これは停電中もずっと動かし続ける装置ではなく、停電が起きたときにシステムを安全に停止させるまでの間、電力を橋渡しする装置だと理解しておきましょう。自家発電設備とあわせて出題されることがあります。また、バックアップの運用では、いつ・何を・どこに・何世代残すかをあらかじめルール化しておくことが重要です。

⚡ ポイント
  • サービスデスク＝一元窓口（SPOC）、解決できなければエスカレーション
  • UPSは「安全に終了させるための橋渡し」であって「停電中もずっと稼働させ続ける装置」ではない、というひっかけに注意

7-8システム監査 — 独立した立場でチェックする

🎯 ゴール：システム監査人の独立性の意味と、監査の一連の流れを説明できるようになる。

システム監査人とは、被監査部門（チェックされる側）から独立した立場で、情報システムを「点検・評価・助言」する人のことです。監査人自身がシステムを直接改善する作業を行うわけではありません。この独立性がなぜ大事かというと、自分の答案を自分で採点したら甘くなってしまうのと同じで、監査される側の人がそのまま監査も行ってしまうと、公正なチェックができなくなるからです。

監査は次のような流れで進みます。監査証拠を集める → 監査調書にまとめる → 指摘事項・改善勧告を出す → 後日、改善されたかをフォローアップする。

✏️ 例題（本試験風） 次のうち、システム監査人の独立性が保てないおそれがあるものはどれか。
(ア) 社長直属の監査室に所属する監査人が、情報システム部門を監査する。
(イ) 外部の監査法人に依頼して監査を行う。
(ウ) 開発を担当したシステム部門の担当者自身が、そのシステムの監査を行う。
(エ) 監査対象部門と業務上の関わりがない総務部門の担当者が監査を行う。
答えを見る 答え：(ウ)
自分が開発したシステムを自分で監査すると、間違いや問題点を指摘しにくくなり、公正な監査ができなくなります。(ア)(イ)(エ)はいずれも監査対象から独立した立場にあるため問題ありません。
⚡ ポイント
  • 「監査人＝第三者的な立場」を崩す選択肢（開発者本人が監査する、監査対象部門の上司が兼務する、など）がひっかけとして出やすい
  • 監査人は改善そのものは行わない、「助言する」役割にとどまる点も覚えておく

7-9内部統制とITガバナンス — 組織としての統制の仕組み

🎯 ゴール：内部統制における職務の分離と、ITガバナンスの意味を説明できるようになる。

内部統制とは、業務が適正に行われるようにするための組織内の仕組みです。代表的なのが「職務の分離」で、たとえば取引を申請する人と承認する人を別の人にする、というルールです。自分でお小遣い帳をつけて自分で承認するのはおかしい、というのと同じで、同じ人が申請と承認の両方を行えてしまうと、不正や誤りに誰も気づけなくなってしまいます。

ITガバナンスは、経営層がIT投資やIT戦略の方向性を決め、それがきちんと実行されているかを統制していく枠組みのことです。「現場任せにせず、経営が主導する」という視点が問われるポイントです。

⚡ ポイント
  • 職務の分離の具体例（申請者≠承認者、開発担当者≠運用担当者など）はパターンを変えて出題される → 「同じ人が矛盾する2つの役割を兼ねていないか」で判定する
  • ITガバナンス＝経営層がITの方向づけと統制を主導する枠組み

答えを見る前に、まず口に出して答えてみるのが定着のコツ。

Q1. 図7-3のアローダイアグラムで、①から⑥までの最短完成日数は何日か。
14日。経路は3つあり、①→②→④→⑤→⑥＝13日、①→②→③→⑤→⑥＝4+3+4+3＝14日、①→③→⑤→⑥＝13日。最大の14日が答え。 → 解説 7-3 に戻る
Q2. 図7-3で、クリティカルパス上の作業D（②→③、3日）が1日遅れると、プロジェクト全体の完成日はどうなるか。
全体も1日遅れる。作業Dは余裕時間（TF）が0のクリティカルパス上の作業なので、遅れがそのまま全体の遅れになる。もし余裕のある作業（C・Fなど、TF＝1）が1日遅れても、その余裕の範囲内なら全体には影響しない。 → 解説 7-3 に戻る
Q3. 24人月と見積もられたプロジェクトを6人で行うと、所要期間は何か月か。
4か月。所要期間＝工数÷人数＝24÷6＝4か月。 → 解説 7-4 に戻る
Q4. システムに障害が起きたとき、まず利用できる状態に戻すことを最優先する活動を何と呼ぶか。
インシデント管理。応急処置にあたり、復旧を最優先する。根本原因を突き止めて再発を防ぐのは問題管理で、両者の役割は異なる。 → 解説 7-5 に戻る
Q5. SLAで稼働率99.99%と定めた場合、年間の停止許容時間はおよそ何時間か。
約0.876時間（約53分）。8,760時間×（1－0.9999）＝8,760時間×0.0001＝0.876時間。稼働率99.9%のとき（約8.76時間）と比べると、一桁上がるだけで許容時間は1/10になる。 → 解説 7-6 に戻る
Q6. システム監査で、監査人の独立性が保てないのはどのような場合か。
監査対象のシステムを開発・運用した本人が、そのシステムの監査を行う場合。いわば「自己監査」になってしまい、公正な点検・評価ができなくなる。 → 解説 7-8 に戻る
Q7. 内部統制における「職務の分離」とはどのようなことか。
たとえば取引の申請者と承認者を別の人にするなど、一人の人物が矛盾する複数の役割（チェックする側とされる側）を兼務しないようにすること。 → 解説 7-9 に戻る
Q8. WBSとガントチャートは、それぞれ何を表すための図か。
WBSは作業を階層的に分解して漏れなく洗い出すための図。ガントチャートは各作業の開始・終了時期を横棒で示し進捗を可視化するための図で、作業間の依存関係は表しにくい。 → 解説 7-2 に戻る
